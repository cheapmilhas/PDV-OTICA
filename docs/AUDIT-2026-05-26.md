# PDV Ótica — Plano de Execução Pós-Auditoria

**Data:** 2026-05-26
**Branch:** main
**Metodologia:** 7 agentes Opus (Security, DB, Backend, Frontend/PDV, Design/UX, Financeiro, OS/Crediário) → 3 revisores (Risk, Business, Feasibility) → síntese
**Artefatos:** `qa-artifacts/audit-2026-05-26/agent-1..7.md` + `reviewer-A/B/C.md`

---

## TL;DR — A verdade em 5 frases

1. **Não está pronto para clientes reais.** Há um backdoor admin ativo (`admin123` + endpoint de seed aberto), receituários médicos em URL público (LGPD Art. 11), e o botão "Finalizar Venda" do PDV aborta a requisição no meio.
2. **2 semanas de trabalho focado** transformam isso em "seguro para primeiros pagantes". **4 semanas** transformam em "SaaS comercial confiante".
3. **O código tem DNA comercial real**: arquitetura multi-tenant moderna, OCR de receita com IA (diferencial único), Kanban de OS, motor de fidelidade — derrotam ssOtica e ControleNaMão em vários eixos.
4. **3 gaps fatais vs concorrência**: NFC-e fiscal (regulatório), WhatsApp automático (table-stakes em 2026), Pix QR no checkout. Sem isso, a venda comercial trava em 60%+ do mercado formal.
5. **Maior ROI da auditoria**: deletar **uma linha** em `modal-finalizar-venda.tsx:215` (`window.location.href`). Fix de 2 minutos que destrava o fluxo crítico de venda crediário.

### SEMANA 2 — Integridade de dados & dinheiro (sem schema migration)

**Objetivo:** zerar bugs financeiros que destruiriam confiança.

#### PR-1A-1: Race condition de cashback (4h)
- Mover check de saldo + débito para dentro da transação com UPDATE condicional atômico (`WHERE balance >= cashbackUsed`)
- Arquivos: `src/services/sale.service.ts:253-268`, `applyCashbackUsageInTx`
- **Atenção:** validar mesmo caminho em `quote.service.ts` (Quote → Sale conversion)

#### PR-1A-2: AR reversal ghost cash (1 dia)
- Criar `CashMovement` compensatório (direction OUT, type ADJUSTMENT) no estorno
- Criar `FinanceEntry` compensatório invertendo PAYMENT_RECEIVED
- Arquivo: `src/app/api/accounts-receivable/route.ts:373-427`
- **Nota:** dados históricos com ghost cash ficam — documentar, planejar backfill script separado

#### PR-1A-3: Mover finance entries para post-commit (2 dias)
- Mover `applyFinanceEntriesInTx` para `applyPostCommitSideEffects`
- Criar tabela `FinanceEntryRetry` (migração aditiva, zero risco)
- Entries falhos vão para retry queue (sem `console.error` silencioso)
- Remover `timeout: 30_000` da transação de venda após mover entries
- **Cuidado deceptivo:** o flow de cancel deleta FinanceEntries; precisa lidar com cancel ocorrendo antes do job post-commit. +1 dia de trabalho real.

#### PR-1A-4: Permission gating no Financeiro (2h)
- `requirePermission("finance.view")` em todas as rotas GET `/api/finance/**`
- `requirePermission("finance.manage")` em POST/PATCH
- ~8 arquivos de rota

**Marco fim da Semana 2:** seguro para primeiros pagantes.

## "DECEPTIVE EASY" — Cuidados de engenharia

Findings que parecem pequenos no relatório mas são maiores na prática:

1. **Wiring `prisma-tenant.ts`** — relatado como 1-2 dias, real 4-7 dias. **Recomendação: pular. ESLint rule entrega 80% da proteção com 5% do esforço.**
2. **OS status machine server-side** — fix de código é 10 min, mas quebra o `APPROVED → IN_PROGRESS` do Kanban imediatamente. Coordenar com VALID_TRANSITIONS no mesmo PR. **1 dia real.**
3. **Card cashDate fix** — código é 1 dia, mas migração de dados históricos + comunicação ao cliente é mais 2-3 dias. **Sempre deferir para Fase 4.**
4. **`console.log` → logger** — 77+ calls. Mecânico não funciona: cada call precisa decisão sobre PII (`email` → `userId`). **1.5 dia real.**
5. **Subscription enforcement** — middleware global precisa whitelist cuidadoso de admin/cron/subscription routes. **2 dias real.**
6. **Dark mode** — toggle é 0.5 dia, mas ~30% das páginas ficam imperfeitas até refactor de tokens. **Aceitar imperfeição inicial.**
