# Plano de Execução — Emissão de Nota Fiscal (NFC-e/NF-e) via Focus NFe

**Data:** 2026-07-05 · **Status:** AGUARDANDO APROVAÇÃO DO DONO — nada implementado.
**Objetivo:** destravar a emissão de NFC-e para as óticas clientes, começando pelo Ceará, com gestão no admin da ótica e monitoramento no super admin.

---

## 1. Resumo executivo

Três descobertas mudam o tamanho e a urgência do projeto:

1. **~40% do backend já existe no código, dormente.** Cliente HTTP Focus NFe (`src/lib/focus-nfe.ts`), rota de emissão (`POST /api/fiscal/nfce/emit`), rota de cancelamento, webhook com HMAC fail-closed, campos fiscais em `Sale` e `Branch`, enum `FiscalStatus`. Tudo desabilitado (sem `FOCUS_NFE_TOKEN` → 503). Falta principalmente: UI, super admin, correções pontuais e a mudança de arquitetura de token (item 3).

2. **O regulatório virou a nosso favor — e criou urgência real no Ceará.** O Decreto 36.417/2025 extinguiu o MFE/CF-e: desde **01/01/2026 a NFC-e é obrigatória para todo o varejo cearense**. Suas óticas clientes **já são obrigadas por lei a emitir NFC-e hoje** — quem não emite está irregular. Em SP, o SAT também morreu (31/12/2025). Consequência estratégica: **NFC-e (modelo 65) + NF-e (modelo 55) cobrem os 26 estados + DF** — não existe mais hardware fiscal a suportar. Um único fluxo serve o Brasil inteiro; a diferença entre estados é só operacional (credenciamento, CSC, prazos), não técnica.

3. **Correção de arquitetura: o token da Focus NFe é POR EMPRESA (CNPJ), não global.** O código atual assume um `FOCUS_NFE_TOKEN` único em env. Na Focus, cada empresa cadastrada recebe `token_producao` + `token_homologacao` próprios; a plataforma usa um token administrativo da conta-mãe para a **API de empresas** (`POST /v2/empresas` — cadastra CNPJ, envia certificado A1 em base64, CSC, habilita NFC-e/NF-e). É o modelo desenhado exatamente para SaaS multi-tenant.

**Recomendação:** executar em 6 fases (~4-6 semanas de trabalho técnico), piloto com o Atacadão (CE), kill-switch por ótica (o `Branch.fiscalEnabled` que já existe, default OFF), e módulo fiscal como feature de plano (nova fonte de receita).

| **NF-e (mod. 55)** | Venda com CNPJ no destinatário, devolução a fornecedor, transferência entre filiais, remessa p/ laboratório de lentes. | Fase 6 |
| **NFS-e** | Conserto cobrado, exame de vista cobrado. Padrão nacional obrigatório desde 01/2026, mas **instável** (inclusive Fortaleza). | Futuro — não prometer no MVP |

⚠️ **Regra nova crítica (vigente desde 05/01/2026):** NFC-e **não pode mais ser emitida para CNPJ** — a SEFAZ rejeita. Cliente pediu "nota no CNPJ" → tem que ser NF-e 55. No MVP, o PDV deve **bloquear com mensagem clara** ("nota para CNPJ será atendida via NF-e — em breve / fale com o suporte"); o fluxo NF-e completo fica para a Fase 6.

### 2.2 Ceará (prioridade)

- **Obrigatoriedade:** NFC-e obrigatória para venda a consumidor final PF desde 01/01/2026, qualquer atividade. Óticas retardatárias estão irregulares → **argumento comercial forte**.
- **Credenciamento (por ótica, feito uma vez):**
  1. Inscrição estadual ativa + situação regular no CGF.
  2. **Certificado digital A1 e-CNPJ** (arquivo .pfx + senha, ~R$ 200/ano — a ótica compra com uma AC).
  3. Credenciar-se como emissor em `nfce.sefaz.ce.gov.br/pages/credenciamento.jsf` (login com o certificado).
  4. Gerar o **CSC** (Código de Segurança do Contribuinte) no portal: menu Serviços → Consultar CSC. A SEFAZ-CE dá CSC de **produção e homologação** (cada um com `id_token` + código).
- **Contingência offline:** CE permite transmitir até o 1º dia útil seguinte (mais folgado que o padrão de 24h).
- **Cancelamento:** assumir o padrão nacional de **30 minutos** (o código atual já valida isso). Depois do prazo: devolução via NF-e de entrada (Fase 6).
- **MEI:** dispensado de nota para PF (emissão facultativa); obrigado para PJ. Na prática, óticas estouram o teto do MEI — a maioria dos tenants será ME/Simples (CSOSN 102).

### 2.3 Outros estados (visão geral)

O layout e os webservices da NFC-e são **nacionais**. O que varia por estado é operacional:

| Variação | Exemplos | Como o sistema trata |
|---|---|---|
| Portal de credenciamento + geração do CSC | Cada SEFAZ tem o seu | Wizard de onboarding com instruções por UF |
| CPF obrigatório acima de um piso | PE ≥ R$ 1.000; PB ≥ R$ 500 (a partir de 05/2026); ES ≥ R$ 10.000; **CE: confirmar** | Tabela de parametrização por UF (`fiscal-uf-rules.ts`) |
| Prazo de contingência | BA/CE/PE/SC: 1º dia útil; demais: 24h | Mesma tabela |
| Prazo de cancelamento | Padrão 30 min; alguns estendem | Mesma tabela |

**Expandir para um novo estado = adicionar uma linha na tabela de parametrização + instruções de credenciamento no wizard.** Zero código novo de integração (a Focus abstrai a comunicação com cada SEFAZ).

### 2.4 Divisão de responsabilidades

| Focus NFe abstrai | Nós (sistema) fornecemos | A ótica/contador fornece |
|---|---|---|
| XML, assinatura, QR Code, comunicação com cada SEFAZ, guarda de XML, DANFE, atualizações de layout (inclusive campos CBS/IBS da reforma tributária) | NCM/CFOP/CSOSN/origem/unidade por item, mapeamento de formas de pagamento, regras de quando pedir CPF, controle de status/retry, UI | Certificado A1 (.pfx + senha), credenciamento na SEFAZ, geração do CSC, classificação fiscal dos produtos (validada com o contador) |

## 3. Arquitetura proposta

### 3.1 Modelo Focus NFe (multi-tenant)

```
Conta Focus NFe (nossa, plano Varejo+)
 ├── token administrativo  → API de empresas (onboarding via super admin)
 ├── Empresa: Ótica A – Matriz (CNPJ X) → token_producao / token_homologacao
 ├── Empresa: Ótica A – Filial (CNPJ Y) → tokens próprios
 └── Empresa: Ótica B (CNPJ Z)          → tokens próprios
```

- Cada **filial com CNPJ próprio = uma empresa na Focus** (certificado + CSC + tokens próprios).
- Onboarding via `POST /v2/empresas`: CNPJ, razão social, endereço, `regime_tributario` (1=Simples, 4=MEI), certificado A1 em base64 + senha, CSC (id_token + código), flags `habilita_nfce`/`habilita_nfe`.
- A Focus **guarda o certificado** e retorna a validade → alertamos renovação em D-30/D-15/D-7.
- Tokens por filial ficam no nosso banco **criptografados** (AES-256-GCM com chave em env, padrão a criar — hoje `fiscalCscToken` está em plaintext na Branch; corrigir junto).
- O certificado .pfx **não fica retido no nosso storage**: upload passa direto para a Focus (base64) e descartamos o arquivo. Menos superfície de risco.

### 3.2 Fluxo de emissão (NFC-e é SÍNCRONA na Focus)

Descoberta importante: ao contrário do que o código atual assume (PENDING + webhook), a **NFC-e é autorizada ou rejeitada na mesma requisição** (poucos segundos). O webhook fica como reconciliação/backup e para a futura NF-e 55 (essa sim assíncrona).

```
Venda COMPLETED no PDV
  → [auto-emissão ligada?] POST /api/fiscal/nfce/emit { saleId }
      → grava fiscalStatus: PENDING (trava dupla emissão, já existe)
      → POST Focus /v2/nfce?ref=sale-<id>   ← ref idempotente (retry seguro, evita rejeição 539)
      → resposta síncrona:
          autorizada → AUTHORIZED + chave + XML/DANFE urls → imprime DANFCE
          rejeitada  → FAILED + mensagem amigável (friendlyError já existe) → operador corrige e re-emite
          timeout    → mantém PENDING → cron de reconciliação consulta status
  → venda NUNCA é bloqueada por falha fiscal (nota fica pendente, painel alerta)
```

- **Cancelamento:** já implementado (30 min, justificativa 15-255 chars) — só ganha botão na UI.
- **Cron de reconciliação** (novo, padrão `CRON_SECRET` + cron-job.org): varre `fiscalStatus: PENDING` antigas → `focusNfe.status(ref)` → resolve. Mata o risco de "venda órfã sem nota".
- **Contingência offline:** fora do MVP (exige numeração local + fila). No MVP, SEFAZ fora do ar = nota fica pendente e o cron re-tenta — o CE dá prazo até o 1º dia útil seguinte, então é aceitável. Documentar essa limitação para o cliente.

### 3.3 Dados fiscais: produto com fallback para filial

Hoje o emit usa NCM/CFOP/CSOSN **padrão da filial** para todos os itens — funciona, mas NCM errado é a causa nº 1 de rejeição (schema 225/778). Proposta:

- **`Product` ganha campos fiscais opcionais:** `cfop`, `origem` (0-8), `unidade` (default UN), aproveitando `ncm`/`cest` que já existem no schema mas não são usados.
- **Fallback:** item sem NCM próprio usa o padrão da filial (comportamento atual preservado).
- **Seed de NCM por tipo de produto** (acelera onboarding; ótica valida com o contador):
  - Armação plástico `9003.11.00` · metal `9003.19.10` · Lentes resina/policarbonato `9001.50.00` · vidro `9001.40.00` · Lentes de contato `9001.30.00` · Óculos de sol `9004.10.00` · Óculos de grau prontos `9004.90.xx` · Estojos `4202.3x`.
- CFOP padrão `5102` (venda dentro do estado); CSOSN `102` (Simples). ST em armações/lentes no CE (→ CSOSN 500 + CEST): **confirmar com contador** — parametrizável por filial.

## 4. Fases de execução

### Fase 0 — Decisões e credenciais (dono + comercial, sem código) · ~1 semana em paralelo
1. **Contratar Focus NFe.** Recomendação: começar no plano **Varejo (R$ 59,90/mês, 1 CNPJ, 500 NFC-e)** para o piloto e migrar para **Varejo+ (R$ 629,90/mês, CNPJs ilimitados, 9.000 NFC-e, R$ 0,06/excedente)** na abertura geral. **Confirmar com o comercial da Focus:** API de empresas inclusa no Varejo+? Modelo de revenda para SaaS? Trial de 30 dias com empresa de teste?
2. **Definir preço do módulo fiscal** para os clientes (decisão comercial — ver §6).
3. **Escolher a ótica piloto** (sugestão: Atacadão, já é piloto de comissão/automove) e alinhar com o contador dela: certificado A1, credenciamento SEFAZ-CE, CSC, classificação fiscal dos produtos.
4. **Confirmar com contador/SEFAZ-CE:** piso de CPF obrigatório no CE; ST em armações/lentes no CE; prazo de cancelamento estadual.

### Fase 1 — Fundação técnica · ~3-5 dias
1. Migração Prisma: tokens Focus por filial (criptografados) + campos fiscais no `Product` (`cfop`, `origem`, `unidade`) + criptografar `fiscalCscToken` existente.
2. Util de criptografia de segredos (AES-256-GCM, chave `FISCAL_SECRETS_KEY` em env).
3. Refatorar `focus-nfe.ts`: token por filial (mantendo env global como fallback p/ homologação) + funções da API de empresas (`createCompany`, `updateCompany`, certificado).
4. **Corrigir bug** do `PAYMENT_METHOD_TO_NFCE` (CHECK/BANK_TRANSFER → CHEQUE/BOLETO/etc.).
5. Adaptar emit para resposta **síncrona** + itens com NCM do produto (fallback filial).
6. `prisma generate` e remover os `as any` das rotas fiscais.
7. Tabela `fiscal-uf-rules.ts` (prazo cancelamento, piso CPF, contingência — começa só com CE).
8. Permissões novas no enum: `fiscal.emit`, `fiscal.cancel`, `fiscal.config` (seed cirúrgico, padrão existente). Feature de plano `FISCAL_NFCE` no `FEATURE_REGISTRY` + gate nas rotas.
9. Teste do webhook Focus (único webhook sem `route.test.ts` — apontado em auditoria) + testes do fluxo síncrono.

### Fase 2 — Onboarding fiscal (super admin + admin da ótica) · ~4-6 dias
**Super admin (`/admin`):**
1. Aba **Fiscal** em `admin/clientes/[id]`: status por filial (não configurado → aguardando certificado → aguardando CSC → homologação → produção), botão "Cadastrar na Focus" (API de empresas), campos IE/regime, toggle `fiscalEnabled` por filial (kill-switch), validade do certificado. Tudo com `getAdminSession()` + `GlobalAudit`.
2. Estender `api/admin/companies/[id]/branches` para os campos `fiscal*`.

**Admin da ótica (`/dashboard/configuracoes/fiscal`):**
3. Substituir o placeholder por um **wizard de credenciamento**: passo a passo SEFAZ-CE (com links), upload do certificado A1 (.pfx + senha → direto para a Focus, arquivo descartado), cadastro do CSC (id + código, homolog. e produção), dados fiscais da filial (IE, regime, CSOSN/CFOP/NCM padrão), seleção de ambiente.
4. Tela de produtos: coluna/edição de NCM + ação em massa "aplicar NCM por categoria" (seed do §3.3).

### Fase 3 — Emissão no PDV e em Vendas · ~4-6 dias
1. `modal-finalizar-venda.tsx`: ao concluir venda com fiscal ligado → emite automático (configurável: automático vs botão), spinner curto, resultado com link DANFE + impressão. Substitui o `NoFiscalBanner`.
2. Rejeição: mensagem amigável (`friendlyError`) + botão "Tentar novamente" após correção. **Venda nunca trava por causa da nota.**
3. CPF na finalização: campo opcional "CPF na nota" (cliente já cadastrado preenche sozinho); aviso de bloqueio para CNPJ (§2.1).
4. Lista/detalhe de vendas: coluna status fiscal (badge), ações Emitir / Re-emitir / Cancelar (30 min, com justificativa) / Ver DANFE / Baixar XML — gated por permissão.
5. Impressão DANFCE térmica (verificar formato retornado pela Focus: HTML vs PDF).

### Fase 4 — Robustez e operação · ~3-4 dias
1. **Cron de reconciliação** de notas PENDING (novo endpoint `api/cron/fiscal-reconcile`, `CRON_SECRET`, cron-job.org 5 min).
2. **Painel "Notas com problema"** no dashboard da ótica (FAILED/PENDING antigas) + contadores.
3. **Alerta de certificado vencendo** (D-30/D-15/D-7): aviso no dashboard + e-mail (fila existente).
4. Inutilização de numeração (endpoint Focus) para gaps de série — ação no super admin.
5. **"Pasta do contador":** export mensal de XMLs por período (feature valiosa; XML deve ser guardado 5 anos — a Focus guarda, mas oferecemos o download).

### Fase 5 — Monitoramento no super admin · ~2-3 dias
1. `admin/fiscal`: visão consolidada — notas por ótica/dia, taxa de autorização vs rejeição, notas presas, certificados vencendo (todas as óticas), consumo da franquia Focus (controle do custo R$ 0,06/nota).
2. Alertas: ótica com rejeições em sequência → sinal de suporte proativo.

### Fase 6 — Expansão (backlog, depois do MVP estável)
- **NF-e 55**: venda para CNPJ, devolução a fornecedor, transferência entre filiais (fluxo assíncrono + webhook, boa parte da infra já pronta).
- **Contingência offline** (numeração local + fila de retransmissão).
- **NFS-e** (conserto/exame) — aguardar estabilização do padrão nacional em Fortaleza.
- **Novos estados**: linha na tabela de UF + instruções no wizard. Sem código de integração novo.

## 5. Rollout (mesmo playbook do motor de comissão)

1. **Homologação** — empresa de teste na Focus, ambiente homologação SEFAZ-CE, suite completa no Preview.
2. **Piloto** — Atacadão em homologação (notas "SEM VALOR FISCAL") por alguns dias → vira produção. `Branch.fiscalEnabled` só nele.
3. **Produção CE** — abrir para as demais óticas do CE, onboarding assistido (wizard + suporte). Migrar plano Focus para Varejo+.
4. **Outros estados** — conforme demanda.

**Kill-switch em 3 camadas:** `Branch.fiscalEnabled` (por filial, default OFF) → feature de plano `FISCAL_NFCE` (por plano/ótica) → env global (remove token master = 503 em tudo, comportamento atual).

## 7. Riscos principais e mitigações

| Risco | Mitigação |
|---|---|
| Certificado A1 vence e emissão para (causa nº 1 de suporte) | Alertas D-30/15/7 + reupload self-service (F4) |
| CSC revogado/errado → rejeição em massa | Wizard valida com emissão de teste em homologação; painel de rejeições |
| Rejeição 539 (duplicidade) em retry | `ref` idempotente por venda (já implementado) |
| NCM errado → rejeição 225/778 | Seed por categoria + validação com contador no onboarding |
| Venda órfã sem nota (timeout/queda) | Status PENDING + cron de reconciliação + painel de pendências |
| SEFAZ-CE fora do ar | Nota fica pendente, cron re-tenta; CE aceita até 1º dia útil. Contingência offline formal fica p/ F6 |

| NFS-e nacional instável | Fora do escopo; não prometer |
| Reforma tributária (CBS/IBS em teste em 2026) | Focus absorve as NTs; schema de produto já nasce extensível |
